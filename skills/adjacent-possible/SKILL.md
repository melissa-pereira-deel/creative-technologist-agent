# Adjacent Possible

At any point, the set of next possible states is constrained by the current state. Each step expands the frontier for the next one. The web succeeded where Xanadu failed not because it was better — but because it was adjacently possible in 1989.

## Model

**First-Order Adjacent Possible** — what you can build THIS MONTH with what you have: skills, tools, users, knowledge.

**Second-Order Adjacent Possible** — what becomes buildable AFTER you build the first-order things. Shipping an MVP creates user feedback, which creates understanding, which makes the next version adjacent.

**The Vision Horizon** — the full product, several orders away. Direction, not destination. Don't abandon it; build toward it through adjacent steps.

**Preconditions** — for any desired feature, ask: what needs to be true first? Technology, users, infrastructure, market conditions. Missing preconditions don't mean "can't" — they mean "work on the precondition."

**Two Traps:**
- *Nelson's Trap* — vision so complete that nothing less feels worth building. Xanadu never shipped. The web did.
- *Incrementalist Trap* — shipping feature after feature with no accumulated direction. Busy but not moving forward.

## Apply

**Adjacent Possible Map** — for your current vision:
1. What is the ultimate goal, unconstrained?
2. What can you build THIS MONTH with what you have?
3. What does building that unlock next?
4. Does the path from 2 → 3 accumulate toward 1?

**Precondition Checklist** — for any desired feature that feels out of reach:
- Technology: do we have it, or can we build it?
- Knowledge: do we understand the problem well enough?
- Infrastructure: database, APIs, distribution?
- Users: does this need N users to be meaningful?
- Market timing: is the environment ready?

**Minimum Viable Step** — for any goal that feels too large:
- What's the smallest step that teaches something real?
- What's the smallest version that delivers user value?
- Can you build it in 1–2 weeks? If not, decompose further.

**The "What Does This Unlock?" Test** — before building anything:
- What becomes possible after this exists that wasn't before?
- Terminal features (useful but unlock nothing) vs. gateway features (open the next order). Prioritize gateways.

**Accumulation Check** — periodically review recent work:
- Does each step expand the frontier toward the vision?
- If you drew a line from each step toward the goal, do the lines converge?

## Anti-Patterns

**"We need to build the whole thing before launching"** — Nelson's Trap. Ship the gateway.

**Building without checking direction** — Incrementalist Trap. Busy ≠ accumulating.

**Treating the adjacent possible as fixed** — it changes with every step you take and everything the world builds.

**Confusing "not yet possible" with "impossible"** — preconditions can be built toward.

## Connections

- **Temporal Reasoning** — the adjacent possible changes over time; pace layers determine when possibilities emerge
- **Evolutionary Design** — each step is a variation tested against reality, expanding the frontier
- **Constraint Propagation** — building X makes Y and Z possible; trace the unlocks
- **Deep Deliberation** — understanding what's adjacent requires deliberation, not just feature listing
- **Economic Model Design** — business model constrains and enables what's adjacent
